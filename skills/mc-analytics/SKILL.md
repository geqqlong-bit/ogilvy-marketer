## 触发条件

当用户提到以下任意情形时使用本技能：

- "数据分析" / "帮我看看这些数据" / "这些数字说明什么"
- "转化率掉了" / "ROI 不好" / "效果变差了" / "数据异常"
- "预算怎么分" / "钱应该花在哪" / "哪个渠道该加预算"
- "A/B 测试结果怎么看" / "这个测试有没有意义"
- "KPI 怎么定" / "指标体系怎么搭"
- "帮我算一下 XX" / "XX 指标是什么意思"
- "为什么 XX 指标在跌" / "XX 涨了但 YY 没涨"
- "预测一下下个月的 XX"

## 与其他技能的关系

| | mc-analytics | mc-report |
|---|---|---|
| 时机 | 随时（事前/事中/事后） | 事后复盘 |
| 输入 | 任意数据片段 | 完整 campaign 执行数据 |
| 输出 | 诊断结论 + 下一步动作 | 结构化复盘报告 |
| 深度 | 单点深挖（一个漏斗、一个渠道、一个测试） | 全局回顾 |

mc-analytics 解决"这个数字什么意思、为什么变了、怎么办"；mc-report 解决"整个 campaign 做得怎么样"。

## 核心原则

**数字不是答案，决策才是。** 所有分析输出必须以"所以你应该..."结尾。

- 营销人给你一堆数字 → 你告诉他哪里有问题、为什么、怎么修
- 营销人不知道该看什么 → 你帮他搭指标体系、定基准线
- 营销人看不懂统计概念 → 你用日常语言翻译，不出现公式
- 营销人需要做判断 → 你给出明确建议，不是"取决于情况"

**翻译规则：**
- 不输出公式，用自然语言解释计算逻辑
- 不说"p值为0.03"，说"这个差异有 97% 的把握是真实的，不是随机波动"
- 不说"置信区间"，说"真实值大概率落在 X 到 Y 之间"
- 不说"统计显著"，说"可以放心采用" 或 "还不能下结论，需要再跑 X 天"
- 所有百分比都给参照物（"3.2% 的转化率，比行业均值 2.1% 高出 52%"）

## 工作方式

1. 确认分析场景（用户要解决什么问题）
2. 读取 campaign 上下文（如有 `brief.md`、`strategy.md`、`channel.md`）
3. 接收用户提供的数据（粘贴数字、截图描述、文件导入）
4. 产出写入 `campaigns/{project-slug}/analytics.md`（如无 campaign 上下文，写入 `analytics/{topic-slug}.md`）

## 模块 1：漏斗诊断

### 适用场景

用户说"转化率低" / "效果不好" / "钱花了但没效果" — 需要定位到底是哪个环节出了问题。

### 通用漏斗模型

```
曝光 → 点击 → 到达 → 浏览 → 加购/留资 → 转化 → 复购
 │       │       │       │         │          │       │
 │       │       │       │         │          │       └─ 复购率
 │       │       │       │         │          └─ 转化率 (CVR)
 │       │       │       │         └─ 加购率 / 留资率
 │       │       │       └─ 页面浏览深度 / 停留时长
 │       │       └─ 到达率（排除跳出）
 │       └─ 点击率 (CTR)
 └─ 曝光量 / 展现量
```

### 诊断流程

**第一步：画出漏斗，标数字**

```markdown
## 漏斗诊断

| 环节 | 数据 | 转化率 | 行业基准 | 判断 |
|------|------|--------|---------|------|
| 曝光 | {数字} | — | — | — |
| 点击 | {数字} | {CTR}% | {基准}% | 🟢正常 / 🟡偏低 / 🔴异常 |
| 到达 | {数字} | {到达率}% | >95% | |
| 浏览 | {数字} | {非跳出率}% | {基准}% | |
| 加购/留资 | {数字} | {加购率}% | {基准}% | |
| 转化 | {数字} | {CVR}% | {基准}% | |
```

**第二步：找到最大断裂点**

漏斗中流失最严重的环节 = 优先修复点。不是每个环节都值得优化，只修最漏的那个桶。

**第三步：归因到具体原因**

每个环节的常见病因：

| 环节异常 | 常见病因 | 排查方向 |
|---------|---------|---------|
| CTR 低 | 素材吸引力不够 / 定向太宽 / 标题无钩子 | 换素材 > 收定向 > 改标题 |
| 到达率低 | 页面加载慢 / 链接失效 / 重定向过多 | 查页面速度 > 检查链接 > 减少跳转 |
| 跳出率高 | 落地页与广告不匹配 / 首屏无价值 / 加载慢 | Message Match > 首屏改造 > 速度优化 |
| 加购率低 | 价格超预期 / 信息不够 / 信任感不足 | 定价策略 > 补充信息 > 增加社会证明 |
| CVR 低 | 结账流程复杂 / 支付方式少 / 运费惊吓 | 简化流程 > 增加支付 > 运费策略 |
| 复购率低 | 无触达 / 产品无复购属性 / 竞品截流 | 购后流程 > 产品评估 > 竞品分析 |

**第四步：输出诊断结论**

```markdown
## 诊断结论

**最大瓶颈：** {环节名} — {具体数字}，远低于行业基准 {基准值}

**根因判断：** {1-2 句话解释为什么这个环节出了问题}

**修复建议（按优先级）：**
1. {立即可做的事} — 预期改善 {X}%
2. {需要 1-2 周的事} — 预期改善 {X}%
3. {需要更长时间的事} — 预期改善 {X}%

**修复后的漏斗预测：**
如果 {环节} 从 {当前值} 提升到 {目标值}，最终转化数从 {当前} → {预测}，增量 {X}%
```

### 场景化漏斗模板

**电商漏斗**
```
广告曝光 → 点击 → 商品页浏览 → 加购 → 发起结算 → 支付成功 → 确认收货 → 复购
```

**私域漏斗**
```
公域曝光 → 点击/扫码 → 加好友/关注 → 激活（首次互动）→ 留资/领券 → 首购 → 复购 → 裂变
```

**B2B 漏斗**
```
内容触达 → 网站访问 → 留资（MQL）→ 销售跟进（SQL）→ 商机 → 提案 → 成单 → 续约/增购
```

**App 漏斗**
```
广告曝光 → 点击 → 应用商店页 → 下载 → 打开 → 注册 → 激活（核心动作）→ 留存 → 付费
```

## 模块 2：归因分析

### 适用场景

用户说"不知道哪个渠道真正带来了效果" / "应该把预算给谁" / "这个渠道到底有没有用"

### 归因模型翻译

不同的归因方式会给出不同的"功劳分配"。帮用户理解每种方式的偏向：

| 归因方式 | 通俗解释 | 偏向 | 适合 |
|---------|---------|------|------|
| 最后点击 | "谁最后推了一把就算谁的" | 高估底层转化渠道（搜索/品牌词） | 预算少、渠道少、起步阶段 |
| 首次点击 | "谁最先让用户知道的就算谁的" | 高估上层认知渠道（社媒/展示广告） | 评估品牌认知投入 |
| 线性归因 | "每个触点平均分功劳" | 无偏向，但也无洞察 | 触点少（<4 个）的简单链路 |
| 时间衰减 | "越靠近成交的触点功劳越大" | 轻微高估底层 | 决策周期长的品类 |
| 数据驱动 | "AI 算每个触点的实际贡献" | 依赖数据量，黑盒 | 月转化 >300 次 |
| Blended ROAS | "别管谁的功劳，总投入除以总产出" | 不看细节，看大盘 | 规模化阶段的宏观指标 |

### 跨渠道归因诊断

```markdown
## 渠道贡献分析

### 各渠道数据

| 渠道 | 花费 | 平台报告转化 | 平台 ROAS | 最后点击转化 | 最后点击 ROAS | 差异 |
|------|------|------------|----------|------------|-------------|------|

### 差异解读

差异大的渠道 = 这个渠道的"助攻"价值与"终结"价值不同

- **平台 ROAS > 最后点击 ROAS**：该渠道在平台的归因窗口内有很多 view-through 转化，实际直接推动力可能没有平台报告那么强
- **平台 ROAS < 最后点击 ROAS**：该渠道的直接推动力被低估，可能是归因窗口设置太短

### 建议的归因策略

{根据用户的渠道组合和阶段，推荐具体的归因方式和解读逻辑}
```

### 增量测试框架

当归因模型"谁都说不清"时，用实验代替模型：

```markdown
## 增量测试建议

**要验证的问题：** {渠道 X} 到底带来了多少增量？

**测试方法：**
- 地理位切割：在 A 区域投放 / B 区域不投放，对比转化差异
- 时间切割：开投 2 周 / 关停 2 周，对比转化变化
- 预算梯度：从 {当前预算} 加到 {+50%}，看转化是否等比增长

**结果判读：**
- 如果关停后总转化几乎不变 → 该渠道增量价值低
- 如果关停后总转化明显下降 → 该渠道有真实增量
- 如果加预算但转化不等比增长 → 已触达效率天花板
```

## 模块 3：预算优化

### 适用场景

用户说"预算怎么分" / "哪个渠道该加钱" / "预算砍了怎么分配"

### 预算分配诊断

```markdown
## 预算分配分析

### 当前分配 vs 效率

| 渠道 | 当前预算 | 预算占比 | ROAS/CPA | 边际效率 | 建议 |
|------|---------|---------|---------|---------|------|

### 边际效率判断

**边际效率** = 最后加的那一块钱带来了多少回报。如果一个渠道继续加预算但 ROAS 在下降，说明它正在"吃饱"。

判断逻辑：
- ROAS 随预算增加而提升或持平 → 🟢 还有空间，可以加
- ROAS 随预算增加而下降但仍 >1 → 🟡 接近天花板，谨慎加
- ROAS 随预算增加跌破 1 → 🔴 已超效率边界，应该减

### 优化建议

**总预算 {X} 的推荐分配：**

| 渠道 | 推荐预算 | 占比 | 调整 | 理由 |
|------|---------|------|------|------|
```

### 预算场景模拟

用户经常面临的三种预算场景：

**场景 A：预算增加**
```markdown
## 加预算优先级

当前总预算 {X}，增加到 {Y}，增量 {Z} 怎么花？

| 优先级 | 渠道 | 分配额 | 理由 |
|--------|------|--------|------|
| 1 | {边际效率最高的渠道} | {金额} | ROAS 仍在上升通道 |
| 2 | {未饱和的渠道} | {金额} | 还有目标人群未覆盖 |
| 3 | {测试新渠道} | {金额，≤增量的 15%} | 小预算测试，寻找新增长点 |
```

**场景 B：预算缩减**
```markdown
## 砍预算优先级

当前总预算 {X}，缩减到 {Y}，需砍 {Z}

| 砍除顺序 | 渠道 | 砍除额 | 理由 |
|---------|------|--------|------|
| 先砍 | {ROAS 最低 / 边际效率最差} | {金额} | 砍掉影响最小 |
| 再砍 | {品牌类 / 长周期渠道} | {金额} | 短期内不产出直接转化 |
| 最后砍 | {核心转化渠道} | {金额} | 砍这里会直接影响收入 |
| 不能砍 | {留存/CRM} | — | 砍 CRM 伤客户终身价值 |
```

**场景 C：零基预算**
```markdown
## 从零搭建预算

总预算 {X}，目标 {Y}

### 阶段 1：验证期（总预算的 20%）
| 渠道 | 预算 | 目标 |
|------|------|------|
| {最快验证的渠道} | | 跑出基准 CPA 和转化率 |

### 阶段 2：放量期（总预算的 50%）
| 渠道 | 预算 | 目标 |
|------|------|------|
| {验证期效率最好的渠道} | | 在保持效率的前提下扩量 |

### 阶段 3：优化期（总预算的 30%）
| 渠道 | 预算 | 目标 |
|------|------|------|
| {再分配到高效渠道 + 测试新渠道} | | 精细化运营 |
```

## 模块 4：A/B 测试分析

### 适用场景

用户说"测试结果怎么看" / "A 方案好还是 B 方案好" / "这个测试要跑多久" / "样本量够不够"

### 测试结果解读

用户提供两组数据，系统直接给判断：

```markdown
## A/B 测试诊断

### 数据摘要

| | 方案 A | 方案 B | 差异 |
|---|--------|--------|------|
| 样本量 | {n_A} | {n_B} | |
| 转化数 | {c_A} | {c_B} | |
| 转化率 | {r_A}% | {r_B}% | {diff}% |

### 结论

**{以下四种结论之一}：**

1. ✅ **B 方案胜出，可以放心切换。**
   B 比 A 高出 {X}%，这个差异有 {Y}% 的把握是真实的。如果全量切换到 B，预计每月多带来 {Z} 个转化。

2. ⏳ **还不能下结论，需要继续跑。**
   目前差异只有 {X}%，要确认这是真实差异而非随机波动，还需要大约 {额外样本量} 个样本（约 {天数} 天）。

3. 🟰 **两个方案没有实质差别。**
   跑了 {天数} 天、{总样本} 个样本后，两者差异只有 {X}%，不值得切换。建议测试更大胆的变量。

4. ❌ **B 方案更差，保持 A 不变。**
   B 比 A 低 {X}%，且有 {Y}% 的把握这不是随机波动。

### 下一步

{根据结论给出具体的下一步动作}
```

### 测试前规划

在跑测试之前帮用户算清楚：

```markdown
## 测试规划

**要测的东西：** {变量描述}
**当前基准：** {当前转化率}%
**想检测的最小差异：** {MDE}%

**需要的条件：**
- 每组至少 {N} 个样本
- 按当前日均流量 {X}，需要跑 {天数} 天
- 测试期间不能调其他变量（否则结果不可信）

**值不值得测？**
- 如果 B 赢了（转化率提升 {MDE}%），每月多带来 {收益估算}
- 测试成本（流量 × 天数 × 可能的效率损失）≈ {成本估算}
- **{值得 / 收益太小不值得花这个时间}**
```

### 常见测试陷阱提醒

用户提供测试数据时，自动检查以下问题：

| 陷阱 | 检测方式 | 提醒语 |
|------|---------|--------|
| 样本量太小 | 每组 <100 或总转化 <30 | "样本太少，现在的差异可能是巧合" |
| 偷看数据 | 用户测试才跑 2 天就来问结果 | "太早了，至少跑满 {建议天数} 天再看" |
| 多变量混淆 | 同时改了 2 个以上变量 | "同时改了多个东西，无法判断是哪个起了作用" |
| 时间段偏差 | A/B 不是同时段跑的 | "不同时间段跑的结果不可比，流量质量可能不同" |
| 分流不均 | 两组样本量差异 >10% | "两组人数差太多，可能分流有问题" |
| 新奇效应 | B 方案前 2 天效果好但在衰减 | "新方案可能有新鲜感加成，建议看第 3 天以后的数据" |

## 模块 5：异常检测

### 适用场景

用户说"XX 突然掉了" / "数据不正常" / "为什么突然涨了" / "今天的数字不对"

### 异常诊断框架

```markdown
## 异常诊断

### 异常描述
- **指标：** {哪个指标}
- **异常方向：** 上涨 / 下降
- **幅度：** {变化百分比}
- **持续时间：** {从什么时候开始}

### 排查清单

按概率从高到低排查：
```

**数据异常（先排除假异常）**

| 检查项 | 状态 |
|--------|------|
| 数据源是否完整（有没有延迟、丢数据） | |
| 统计口径是否变了（平台更新了统计方式） | |
| 标签/UTM 是否正确（有没有标错渠道） | |
| 对比基准是否合理（不要拿周末比工作日） | |

**外部因素**

| 检查项 | 状态 |
|--------|------|
| 行业/品类是否有整体变化（不只是你在变） | |
| 竞品是否有大动作（打价格战、大促、爆款） | |
| 平台是否有规则/算法变化 | |
| 是否有季节性/节假日因素 | |
| 是否有社会事件影响（新闻、政策） | |

**内部因素**

| 检查项 | 状态 |
|--------|------|
| 是否有投放变更（素材、出价、定向、预算） | |
| 是否有页面变更（落地页、产品页、价格） | |
| 是否有产品变更（库存、定价、SKU 上下架） | |
| 是否有技术问题（页面加载慢、链接坏、追踪代码丢失） | |

```markdown
### 诊断结论

**最可能的原因：** {具体原因}

**判断依据：** {为什么排除了其他原因，指向这个原因}

**应对建议：**
- 如果是 {原因 A} → {动作}
- 如果是 {原因 B} → {动作}

**是否需要紧急处理：** 是 / 否
- 是 → {立即要做的事}
- 否 → {持续观察 X 天，如果 {条件} 再处理}
```

### 常见异常模式识别

| 模式 | 特征 | 常见原因 | 建议 |
|------|------|---------|------|
| 断崖下跌 | 某天突然掉 >50% | 技术故障 / 账户问题 / 追踪丢失 | 先查技术层面 |
| 持续阴跌 | 每周下降 5-10%，持续 3 周+ | 素材疲劳 / 竞争加剧 / 季节性 | 换素材 > 调策略 |
| 波动加大 | 日均没变但每天大起大落 | 出价策略变化 / 预算波动 | 检查投放设置 |
| 量跌价涨 | 转化量少了但客单价高了 | 低质流量减少（可能是好事） | 看总收入而非单一指标 |
| 量涨质跌 | 流量多了但转化率跌了 | 扩量带来低质流量 | 收紧定向或降预算 |
| 突然飙升 | 某天暴涨 >200% | 爆款内容 / 被大号转发 / 数据异常 | 确认是否可持续 |

## 模块 7：趋势预测

### 适用场景

用户说"下个月大概能做到多少" / "按这个趋势年底能到多少" / "目标能不能完成"

### 预测方法

**方法 1：趋势外推（最简单）**

基于最近 N 个数据点的趋势，推断下一个周期。

```markdown
## 趋势预测

**数据：** {用户提供的历史数据}

**趋势判断：** {上升 / 平稳 / 下降}，近 {N} 期平均环比 {X}%

**预测：**
- 乐观（趋势加速）：{数字}
- 基准（趋势延续）：{数字}
- 保守（趋势减缓）：{数字}

**可信度：** {高/中/低}
- {高}：数据点 >6 个，趋势稳定，无重大外部变量
- {中}：数据点 3-6 个，或有已知的季节性/外部影响
- {低}：数据点 <3 个，或即将有大促/淡季/竞品动作等干扰

**会影响预测准确性的因素：**
1. {具体因素}
2. {具体因素}
```

**方法 2：目标倒推**

用户有目标但不确定能不能达成：

```markdown
## 目标可行性分析

**目标：** {用户的目标，如"Q2 收入 500 万"}

**当前基础：**
- 当前月均：{X}
- 当前增速：{X}%/月

**达成路径：**
要完成目标，需要从 {当前值} 增长到 {目标值}，意味着：
- 需要月均增速 {Y}%
- vs 当前增速 {X}%
- 差距：每月需要额外 {Z}

**可行性判断：** {容易达成 / 有挑战但可行 / 非常困难 / 基本不可能}

**需要的条件：**
1. {具体条件}（如：广告预算从 X 提升到 Y）
2. {具体条件}（如：转化率从 X% 提升到 Y%）
3. {具体条件}

**风险提醒：**
- {可能导致目标无法达成的因素}
```

**方法 3：季节性修正**

对有明显季节性的品类，在趋势外推基础上加入季节系数：

```markdown
## 季节性预测修正

**品类季节性特征：**
| 月份 | 季节系数 | 原因 |
|------|---------|------|
| 1月 | {系数} | {如：年货节 / 新年促销} |
| ... | | |

**修正后预测：**
- 基准趋势预测：{X}
- 季节系数：{Y}
- 修正后：{X × Y}
```

## 模块 8：营销计算器

### 适用场景

用户说"帮我算一下" / "XX 是多少" — 快速出答案，不需要完整分析。

### 常用计算

**获客效率类**

| 要算的 | 需要的数据 | 计算逻辑 |
|--------|-----------|---------|
| CPA（获客成本） | 总花费、获客数 | 总花费 ÷ 获客数 |
| ROAS（广告回报率） | 广告收入、广告花费 | 广告带来的收入 ÷ 广告花费 |
| ROI（投资回报率） | 利润、总投入 | (收入 - 总成本) ÷ 总投入 |
| CPM（千次曝光成本） | 总花费、曝光量 | 总花费 ÷ 曝光量 × 1000 |
| CPC（单次点击成本） | 总花费、点击数 | 总花费 ÷ 点击数 |

**转化效率类**

| 要算的 | 需要的数据 | 计算逻辑 |
|--------|-----------|---------|
| CVR（转化率） | 转化数、访问量 | 转化数 ÷ 访问量 |
| AOV（客单价） | 总收入、订单数 | 总收入 ÷ 订单数 |
| Cart Abandonment（弃购率） | 加购数、完成支付数 | (加购 - 支付) ÷ 加购 |
| 跳出率 | 只看一页就走的访问、总访问 | 单页访问 ÷ 总访问 |

**客户价值类**

| 要算的 | 需要的数据 | 计算逻辑 |
|--------|-----------|---------|
| LTV（客户终身价值） | AOV、购买频次、客户生命周期 | 客单价 × 年购买次数 × 平均客户年限 |
| LTV:CAC | LTV、CPA | LTV ÷ CPA（健康值 >3x） |
| 复购率 | 复购客户数、总客户数 | 有 2 次及以上购买的客户 ÷ 总客户 |
| CAC Payback | CPA、月毛利/客户 | CPA ÷ 月均客户贡献毛利（得出月数） |

**内容效率类**

| 要算的 | 需要的数据 | 计算逻辑 |
|--------|-----------|---------|
| 互动率 | 互动量、曝光量 | (点赞+收藏+评论+转发) ÷ 曝光 |
| 完播率 | 完整观看数、播放量 | 完整观看 ÷ 开始播放 |
| 粉丝转化率 | 新增粉丝、内容曝光 | 新增粉丝 ÷ 内容总曝光 |
| CPE（单次互动成本） | 总花费、互动量 | 总花费 ÷ 总互动 |

### 快速计算输出格式

```markdown
## {计算名}

**你给的数字：** {复述用户提供的数据}
**结果：** {计算结果}
**解读：** {一句话说明这个数字好不好} — {对比行业基准或历史数据}
**建议：** {如果数字不理想，一句话说怎么改善}
```

## 模块 9：数据看板设计

### 适用场景

用户说"帮我设计个看板" / "日报/周报应该看什么" / "我应该每天盯什么数字"

### 看板分层

```markdown
## 数据看板设计

### 日看板（执行者，<5 分钟看完）

| 指标 | 昨日 | 7 日均值 | 环比 | 状态 |
|------|------|---------|------|------|
| {3-5 个核心执行指标} | | | | 🟢🟡🔴 |

**日看板规则：**
- 只放可以当天行动的指标
- 红灯 = 需要今天处理
- 黄灯 = 需要关注，连续 3 天黄灯升级为红灯

### 周看板（负责人，<15 分钟看完）

| 指标 | 本周 | 上周 | 环比 | 月目标进度 |
|------|------|------|------|-----------|
| {5-8 个一级/二级指标} | | | | |

**周看板规则：**
- 看趋势而非单点
- 对比周环比（消除日波动）
- 标注月目标完成进度

### 月看板（管理层，<30 分钟看完）

| 指标 | 本月 | 上月 | 环比 | 同比 | 年目标进度 |
|------|------|------|------|------|-----------|
| {北极星 + 一级指标} | | | | | |

**月看板规则：**
- 看大盘健康度
- 标注关键决策点（需要加预算/调方向/换策略）
- 附上下月预测
```

### 报警规则

```markdown
## 报警阈值设置

| 指标 | 黄灯阈值 | 红灯阈值 | 响应要求 |
|------|---------|---------|---------|
| {指标} | {偏离基准 X%} | {偏离基准 Y%} | {谁在多久内响应} |

**报警原则：**
- 报警太多 = 没有报警（避免 alert fatigue）
- 红灯指标控制在 5 个以内
- 每个红灯必须有明确的响应人和 SOP
```

## 输出模式

### 快速计算

用户给一个数字让你算 → 直接给结果 + 一句话解读，不超过 5 行。

### 单点诊断

用户问"XX 为什么掉了" / "这个测试结果怎么看" → 只输出对应模块的诊断，控制在 1 页以内。

### 完整分析

用户明确要"做个全面的数据分析" → 按需组合相关模块，输出完整报告。

### 体系搭建

用户要"帮我搭指标体系" / "设计看板" → 输出指标金字塔 + 看板设计 + 报警规则。

---

## 交互规则

- 用户粘贴一堆数字 → 先识别是什么指标、什么场景，再分析，不要让用户解释格式
- 数据不完整 → 先基于已有数据给初步判断，再告诉用户"如果补充 XX 数据可以更准确"
- 用户问指标含义 → 输出指标定义卡，不是百科解释，是"这个数字对你意味着什么"
- 所有建议必须具体可执行 → 不是"提升转化率"，是"PDP 页面增加 3 条以上用户评价，预期 CVR 提升 0.3-0.5pp"
- 承认不确定性 → 预测和诊断都标注可信度，不装确定
- 输出完成后告诉用户文件路径，不要在对话中重复全文
- **严禁凭空编造基准数据。** 引用行业基准时标注来源和适用条件，无数据时说"需要根据你的历史数据建立基准"
